# 多 Agent 复盘与角色迭代

- 视频：纳瓦尔宝典 - 学会销售，学会构建
- 日期：2026-05-18
- 成片版本：v4 final
- 成片评分：9.1 / 10
- 目标：结合本次每个 Agent 的输入与输出，迭代角色定义，让下一条同系列视频默认达到本轮水准。

## 总体结论

多 Agent 系统能正确运行。每个 Agent 都在限定写入范围内产出明确文件，主编总控完成跨阶段整合、渲染、Notion 同步和质量闭环。最大改进点是：系统已经从“单人凭感觉做视频”升级成“每一步有输入、输出、闸口、复盘”的生产线。

本轮的主要系统风险也很清楚：

- 视觉 Agent 如果不检查模板版本，容易在旧 72 秒模板上继续制作。
- 配音与画面时长必须由主编或视觉 Agent 做硬同步，否则结尾会被裁掉。
- 配音导演和视频评审如果不能实际听音频，必须明确评估边界，不允许假装完成主观听感评审。
- Notion 同步属于 G5，不应被评审 Agent 提前勾选。

## G6 追加迭代：状态源与协作契约

这次复盘后又发现 4 个高确定性系统改进点，已固化到系列 SOP、质量标准和角色定义：

- `quality-check.md` 改为主编总控专属状态源，其他 Agent 只写 handoff 或独立评估文件。
- G2 / G4 音频判断拆成 `technical_audio_pass` 和 `audio_subjective_review`；系统没有实际听音频时，主观听感必须保持 pending。
- G5 新增 `notion-sync-receipt.md`，作为 Notion 同步状态的唯一回执。
- 脚本与视觉之间新增 `scene-contract.json`，视觉和配音基于同一份结构化协议协作。

本轮属于“需要修改”的 G6：这些问题已经影响状态一致性和多 Agent 协作可靠性，不是为了形式感强行迭代。

## G6 追加迭代：装配节奏与平台适配

在导演视角复盘后，又确认了 4 个值得固化的系统改进点：

- 新增 `08 装配剪辑 Agent`，在 G3.5 负责样片或等价片段抽检，专门发现“脚本、视觉、配音各自没问题，但拼在一起不顺”的问题。
- `scene-contract.json` 升级到 v1.1，每个 scene 新增 `scene_energy` 和 `retention_goal`，让脚本、视觉、配音和装配按同一套节奏协议协作。
- 选题和 brief 增强平台适配：01 必须输出平台钩子假设和封面误区；02 必须输出观众默认误判和情绪落点。
- 评审和复盘增强传播闭环：06 固定拆成结构审和成片审；07 必须输出下条继续打的点、必须避免的点和值得进入选题池的评论原话。

本轮同样属于“需要修改”的 G6：问题不只是单条视频审美偏好，而是会长期影响同系列视频的节奏装配、平台留存和规模化稳定性。

## 00 主编总控 Agent

输入：Notion 入口、系列 SOP、Agent 角色库、选题库、视频台账、上游所有生产文件。

输出：项目立项、任务拆解、Agent 派发、配音生成、渲染、封面、Notion 同步、最终状态。

表现：能发现新项目仍是旧模板，并要求视觉 Agent 按 v4 标杆升级；能发现 `98.8s` 画面时长与 `105.216s` 配音不一致，并要求修正。

迭代：

- 增加“模板版本检查”：新项目创建后必须确认 HyperFrames 是否为当前 v4 系列模板。
- 增加“音画时长硬检查”：配音完成后，视觉时长必须贴近音频时长，误差建议小于 0.3 秒。
- 增加“G5 收尾职责”：主编负责 Notion 字段、视频页面正文、发布复盘库和本地 `quality-check.md` 的一致性。

## 01 选题研究 Agent

输入：当前系列定位、选题方向、质量标准、目标观众焦虑。

输出：`content-review.md`、`agent-handoffs/01-topic-researcher.md`。

表现：内容评分 9.1，明确“只会做是产能，只会卖是承诺”的核心判断；小林案例有具体人物、数字和对照组。

迭代：

- 必须提供一个可视化证据对照，例如 `17 访问 / 0 付费` 对 `99 元 / 10 单`。
- 必须写清“不能讲成什么”，本轮有效避免了书摘、销售技巧课和创业课。
- 内容评分不仅给总分，还要说明反常识、痛点、案例、观点锋利度四项中的高分依据。

## 02 内容主编 Agent

输入：`content-review.md`、质量标准、系列标杆。

输出：`brief.md`、`agent-handoffs/02-content-editor.md`。

表现：brief 明确了目标观众、痛点、反直觉判断、不讲范围、案例和视觉方向，为脚本和视觉减少了歧义。

迭代：

- brief 必须定义“唯一核心观点”和“不讲什么”，避免后续脚本发散。
- brief 必须给视觉证据方向，而不只是内容结构。
- brief 必须明确发布目标，如本轮的“测试 AI 时代个体价值链方向”。

## 03 脚本 Agent

输入：`content-review.md`、`brief.md`、01/02 handoff、质量标准。

输出：`script.md`、`voice/narration.txt`、`agent-handoffs/03-scriptwriter.md`。

表现：输出 48 句口播，前 3 秒现实冲突明确；`voice/narration.txt` 干净可直接 TTS；第 35 句偏长但在 handoff 中主动提示视觉拆分。

迭代：

- 必须给至少 2 个钩子，并写明最终选择理由。
- `voice/narration.txt` 必须只包含最终口播，不允许 Markdown、括号说明或制作注释。
- 必须标出长句风险，并给视觉/配音拆分建议。
- 必须提供屏幕文字，不让视觉 Agent 从零猜重点。

## 04 视觉导演 Agent

输入：`script.md`、`brief.md`、03 handoff、v4 标杆 HyperFrames。

输出：`shot-list.md`、`hyperframes/index.html`、`hyperframes/content.js`、`agent-handoffs/04-visual-director.md`。

表现：把旧 72 秒模板升级为 v4 结构，补齐系列标识、进度线、案例双栏、诊断卡、行动清单和烧录字幕；lint 和 inspect 均通过。

迭代：

- 开工前必须检查当前项目模板是否落后于标杆版本，落后则先升级模板。
- `content.js meta.duration`、`timing.outroEnd`、`index.html data-duration` 必须与最终音频时长一致。
- 必须输出关键帧抽检建议，覆盖开头、原则、案例左右对照、行动、结尾。
- 案例段必须有场景证据，不允许只有抽象金句卡。

## 05 配音导演 Agent

输入：`script.md`、`voice/narration.txt`、试音、全片配音、VTT 时间轴。

输出：`voice/voice-manifest.json`、G2 状态建议、`agent-handoffs/05-voice-director.md`。

表现：记录了试音和全片参数，明确没有实际听音频，给出基于技术/时间轴的 8.7 预估，并要求 G4 人工抽听。

迭代：

- 如果未实际听音频，必须明确评估边界，不允许写成主观听感结论。
- 必须输出 `voice-manifest.json`，记录 voice、rate、duration、subtitle、tone 和复用建议。
- 不再直接写 `quality-check.md`，G2 只提供状态建议，由主编总控汇总。
- 必须给视觉同步建议，尤其是长句拆分和关键停顿。
- 试音速度和全片速度可以不同，但必须解释原因。

## 06 视频评审 Agent

输入：最终成片、无声版、封面、关键帧、上游全部 handoff、技术校验结果。

输出：`review.md`、G4 状态建议、`agent-handoffs/06-video-reviewer.md`。

表现：给出 9.1 评分，无硬性必改；明确无法实际听音频，不冒充人耳抽听；没有提前勾选 G5 Notion 同步。

迭代：

- 评审必须先写边界：哪些是实际验证，哪些只是技术/文件检查。
- 必须检查关键帧，而不只看脚本和理论结构。
- G4 只能判断成片是否可发布，不能替代 G5 发布同步。
- 不再直接写 `quality-check.md`，G4 只提供评审结论和状态建议。
- 如果没有硬性必改，也必须给发布后数据验证重点。

## 07 发布复盘 Agent

输入：视频台账、发布复盘库、发布资产、平台数据。

输出：发布复盘记录、新选题建议、标题/封面/钩子改进。

表现：本轮已创建待发布复盘记录，但还没有平台数据。

迭代：

- 视频通过 G4 后就创建“待发布”复盘记录，避免发布后忘记回填。
- 发布后必须按平台回填播放量、完播率、点赞率、收藏率、评论数和高价值评论。
- 必须把评论中的高价值反驳或共鸣转入素材引用库或新选题。

## 08 装配剪辑 Agent

输入：`scene-contract.json`、`shot-list.md`、`hyperframes/content.js`、`voice/voice-manifest.json`、关键帧、样片或等价片段抽检。

输出：`assembly-check.md`、G3.5 状态建议、需要退回脚本/视觉/配音的具体修改点。

表现：补齐本集 `assembly-check.md`，确认 S01/S05 强能量段、S03/S04 案例证据段、S02/S06 承接与行动段的节奏梯度成立；未发现硬性音画错位、字幕抢拍、转场打断或 BGM 压口播风险。

迭代：

- 必须按 `scene_id` 逐段记录切点、字幕落字、停顿、转场、BGM/ducking、`scene_energy` 和 `retention_goal`。
- 必须区分结构问题和执行问题：结构问题退回脚本/brief，执行问题退回视觉/配音/HyperFrames。
- 必须给平台节奏风险，例如前 3 秒不够平台语言、案例段拖慢、结尾没有截图或评论触发点。
- 不直接写 `quality-check.md`；只给主编总控 G3.5 状态建议。

## 下一条视频默认要求

- 必须先跑多 Agent 自检。
- 每个 Agent 必须写 `agent-handoffs/xx-role.md`，记录输入、输出、判断、风险和下一步。
- 脚本 Agent 必须生成 `scene-contract.json` v1.1，包含 `scene_energy` 和 `retention_goal`，主编检查后视觉和配音才能进入正式并行。
- 视觉导演必须标注每段制作成本，且每个 scene 只保留一个主信息，数字证据同屏不超过 3 组。
- 配音导演必须输出 `pronunciation_notes`、BGM / ducking 风险和句级别危险点。
- 最终评审前必须先完成 G3.5：样片或等价片段抽检，并生成 `assembly-check.md`。
- 主编必须在渲染前确认模板版本、音频时长、画面时长、字幕覆盖。
- G4 通过后，G5 必须同步 Notion 视频页面、选题库、视频台账和发布复盘库，并生成 `notion-sync-receipt.md`。
- 发布前人工完整看听一遍保留在发布清单中，不作为自动生产流程阻塞项。
